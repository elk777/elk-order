# 偏爱厨房项目简历介绍

> 适用场景：5 年前端工程师简历、2026 年前端岗位中“AI 经验 / AI Coding / Vibe Coding / 全栈协作 / 小程序工程化”项目描述。

## 一、简历可直接粘贴版本

### 项目名称

偏爱厨房｜情侣互动厨房小程序（Vibe Coding / AI 协同开发实践）

### 项目描述

偏爱厨房是一款面向情侣和亲密关系的互动厨房小程序，围绕“饲养员”和“吃货”两种角色，提供情侣绑定、共享厨房菜谱、购物车点单、订单状态流转、每日签到、积分体系、首页皮肤/壁纸等功能。项目采用 uni-app + Vue 3 实现微信小程序前端，并配套 NestJS + Prisma + MySQL + Redis 后端服务。该项目是个人 Vibe Coding 实践项目，重点验证 AI 辅助下的需求分析、任务拆解、代码生成、影响面审查、前后端联调和质量把控能力。

### 技术栈

uni-app、Vue 3、Composition API、Pinia、uview-plus、SCSS、微信小程序、NestJS、Prisma、MySQL、Redis、TypeScript、Docker、AI Coding / Vibe Coding

### 职责描述

- 独立负责小程序前端架构与核心业务开发，完成首页、菜单、菜谱、购物车、订单、我的、签到积分、首页皮肤等主要模块。
- 设计“饲养员 / 吃货”双角色交互模型，实现不同角色下的菜单管理、点单、接单、烹饪、完成订单等差异化流程。
- 设计并落地“共享厨房”业务模型，绑定情侣后合并双方菜谱可见范围，同时保留创建者管理权限，保证菜谱浏览、购物车添加、订单提交链路一致。
- 封装前端请求层，统一处理接口地址、Token 注入、登录过期、错误提示、GET 重试、文件上传等通用能力。
- 联调配套 NestJS 后端服务，参与用户、情侣关系、菜谱、购物车、订单、积分等 Prisma 数据模型和接口契约梳理。
- 使用 AI 工具辅助进行需求澄清、方案对比、代码阅读、影响面分析、初版实现、Bug 定位、重构建议和文档沉淀。
- 建立“AI 输出不直接落地”的质量流程，通过人工 Review、项目风格校对、跨层链路验证、类型检查、构建检查和真机调试把控最终质量。

### 项目难点

- 双角色 + 情侣绑定导致前端视图、后端查询范围和订单操作权限高度耦合，需要统一处理不同用户状态下的业务边界。
- 共享厨房涉及菜谱可见性、创建者权限、购物车添加、订单提交、解绑后恢复隔离等多层逻辑，不能只在前端做展示合并。
- 小程序真机环境下存在接口地址、安全区、自定义导航、弹窗遮罩、文件上传等细节，需要结合真实设备反复验证。
- AI 辅助开发提升了实现速度，但也容易产生“局部代码正确、整体链路不一致”的问题，需要建立明确的 AI 工作流和人工验收标准。

### 项目亮点

- 将 AI Coding 从“生成代码”扩展到需求分析、任务拆解、代码影响面分析、跨层联调、质量检查和文档沉淀，形成完整的 Vibe Coding 实践闭环。
- 在共享厨房功能中，通过 AI 辅助梳理前端菜单、后端菜谱、购物车和订单的调用链，最终由后端统一收口菜谱可见性和下单权限，降低前端绕过风险。
- 前端采用模块化页面、Pinia 状态管理、自定义请求封装和小程序真机适配，兼顾业务体验和工程可维护性。
- 项目覆盖前端体验、后端接口、数据库模型、Redis、Docker 本地联调和小程序真机验证，体现前端工程师对全链路交付的理解。

### 3. 关键业务模型

#### 双角色模型

- `userType = 0`：饲养员
- `userType = 1`：吃货

不同角色在同一模块下有不同操作：

- 饲养员：创建菜谱、编辑菜谱、处理厨房订单、接单、开始烹饪、完成订单。
- 吃货：浏览菜单、加入购物车、提交订单、查看我的订单、取消订单。

#### 共享厨房模型

共享厨房不是简单把双方菜谱在前端拼接，而是在后端统一定义“当前用户可见的厨房范围”：

- 未绑定：菜谱查询范围为当前用户。
- 已绑定：菜谱查询范围为当前用户 + 绑定对象。
- 创建菜谱：仍归属当前用户。
- 编辑/删除菜谱：只允许创建者操作。
- 加购物车/下单：后端校验菜谱是否属于当前共享厨房。

这样可以避免“前端看不到但接口能下单”的越权问题，也让解绑后的数据隔离更自然。

#### 订单状态模型

订单状态由后端统一维护：

- 待接单
- 已接单
- 烹饪中
- 已完成
- 已取消

前端根据当前角色和订单状态展示可用按钮，后端负责最终状态流转校验。

## 五、Vibe Coding / AI 工作流

这部分是整份文档最适合突出 AI 经验的内容。

### 1. 需求澄清：把一句需求拆成可执行任务

在项目中，很多需求最初都是自然语言，例如：

> 两个用户绑定后，他们分别创建的菜谱应该组合成一个厨房，双方都能看到和下单。

AI 参与方式：

- 先让 AI 总结需求目标、用户角色、业务边界和可能影响的模块。
- 要求 AI 区分“可从代码推导的信息”和“需要产品决策的问题”。
- 将需求拆成前端展示、后端查询、权限校验、数据恢复、验收标准等子任务。

人工把控点：

- 判断哪些是 MVP 必须做，哪些是后续扩展。
- 决定是否引入新数据表，还是复用现有情侣关系动态计算厨房范围。
- 明确编辑/删除权限是否开放给对方，避免 AI 过度扩展。

### 2. 代码阅读：先理解现有系统再改代码

在动手前，先让 AI 阅读相关文件，而不是直接生成实现：

- 前端菜单展示：`pages/cart/component/CateTab.vue`
- 菜谱接口封装：`api/recipes.js`
- 购物车状态：`stores/recipe.js`
- 后端菜谱服务：`recipes.service.ts`
- 后端购物车服务：`cart.service.ts`
- 后端订单服务：`orders.service.ts`
- Prisma 数据模型：`schema.prisma`

AI 参与方式：

- 让 AI 汇总当前数据流。
- 标记哪些接口按单用户查询。
- 找出可能绕过前端展示的后端入口。

人工把控点：

- 判断 AI 的影响面是否完整。
- 主动补充遗漏路径，例如“订单创建时传 cartList 也要校验菜谱归属”。
- 避免只改 UI，不改后端权限。

### 3. 方案设计：让 AI 给选项，但由人做决策

共享厨房功能中可以有多种方案：

- 新增 `Kitchen` 表。
- 绑定后迁移双方菜谱到同一个厨房。
- 不改表结构，按当前情侣关系动态计算厨房成员。

AI 参与方式：

- 对比实现成本、数据迁移成本、解绑语义和后续扩展空间。
- 给出推荐方案和风险。

人工把控点：

- 选择 MVP 阶段更稳妥的动态作用域方案。
- 不因为 AI 建议“架构化”就过早加表。
- 保持实现贴合当前代码结构。

### 4. 代码生成：AI 写初稿，人负责边界和风格

AI 可以快速生成重复性的代码，例如：

- 后端 `resolveKitchenUserIds` 作用域查询。
- 前端 `canManage` 控制编辑/删除入口。
- README、PRD、简历项目文档。

但 AI 初稿不会直接合并。

人工 Review 重点：

- 是否符合当前项目命名和代码风格。
- 是否误改用户已有代码。
- 是否改变了无关模块。
- 是否遗漏边界条件。
- 是否能通过 typecheck / build / 真机验证。

### 5. 联调验证：不把“AI 说可以”当结果

每次实现后，会按功能类型进行验证：

- 前端文档：`git diff --check`
- 后端服务：`pnpm typecheck`、`pnpm build`
- 数据清理：Docker MySQL / Redis 命令后做只读验证
- 小程序体验：微信开发者工具和真机预览
- 跨层链路：从页面操作到接口、数据库、状态回写逐层核对

这种流程能避免 AI 代码“看起来对，但业务链路不闭环”的问题。

### 6. 文档沉淀：把 AI 协作过程变成资产

项目中不只写代码，也沉淀了：

- 前端 README
- 后端 README
- 共享厨房 PRD
- 项目简历材料
- Trellis 任务记录
- 项目规范和检查清单

这体现了 AI 工作流的另一个价值：把一次问题排查或一次功能实现，转化成后续可复用的项目知识。

## 七、项目难点

### 1. 双角色和绑定关系导致业务状态复杂

同一个页面在不同角色下展示和操作完全不同。菜单页、订单页、我的页都需要根据 `userType`、登录状态、绑定状态、订单状态动态变化。

解决方式：

- 将用户状态放入 Pinia 统一维护。
- 将订单动作配置化，前端按角色和状态渲染按钮。
- 后端对订单状态流转进行最终校验，避免前端状态异常造成非法操作。

### 2. 共享厨房需要跨前后端统一权限边界

如果只在前端展示双方菜谱，但购物车和订单接口没有同步校验，就可能出现通过接口下单非共享菜谱的问题。

解决方式：

- 后端统一解析当前用户的厨房范围。
- 菜谱列表、分类列表、菜谱详情、购物车添加、订单创建使用同一作用域。
- 前端通过 `canManage` 判断是否展示编辑/删除入口。

### 3. 小程序真机调试细节较多

真机环境下会遇到接口地址、遮罩层高度、刘海屏安全区、图片上传、视频皮肤等问题。

解决方式：

- 使用 `BASE_URL` 支持局域网和内网穿透地址切换。
- 封装安全区和自定义导航高度工具。
- 对弹窗遮罩、首页布局、底部 TabBar 和滚动区域做真机适配。

### 4. AI 代码质量需要主动把控

AI 可以提升速度，但也容易出现“局部代码可用、整体链路不一致”的问题。

解决方式：

- 每次修改前先让 AI 阅读相关代码和调用链。
- 对跨层功能先明确数据流，再写代码。
- 修改后执行类型检查、构建检查、真机验证或接口验证。
- 将踩坑点沉淀到 README、任务文档或项目规范中。
