### 技术栈

uni-app、Vue 3、Composition API、Pinia、uview-plus、SCSS、微信小程序、NestJS、Prisma、MySQL、Redis、TypeScript、Docker、AI Coding / Vibe Coding、Trellis 任务工作流

### 职责描述

- **产品与业务设计**：独立完成产品需求设计、业务建模和功能拆解，覆盖角色模型、情侣绑定、共享厨房权限、订单状态机、积分体系等核心业务。
- **Vibe Coding 全栈交付**：通过自然语言需求表达，引导 AI 生成前端页面、Pinia 状态、HTTP 封装、NestJS Controller / Service、Prisma 数据模型、Redis 缓存等前后端代码，完成一个真实可运行的全栈系统。
- **AI 输出质量把控**：建立"需求澄清 → 影响面分析 → AI 生成初稿 → 人工 Review → 联调验证 → 文档沉淀"六阶段工作流，对 AI 生成的每一段代码进行风格、边界、跨层一致性 Review，避免"局部正确、整体不一致"。
- **技术决策与架构取舍**：在 AI 给出多种实现方案时（如共享厨房是否新增数据表、是否动态作用域），由我做最终架构选型，确保贴合 MVP 范围和现有代码结构。
- **跨前后端联调验证**：通过 Docker、本地 MySQL/Redis、微信开发者工具和真机预览，完成接口契约、Token 流转、订单状态、上传链路、安全区适配等端到端验证。
- **工程规范与文档沉淀**：维护前后端 README、共享厨房 PRD、Trellis 任务记录和项目规范，把 AI 协作过程转化为可复用的项目知识。

### 项目难点

- **模糊需求 → AI 可执行任务的转化**：自然语言需求需要被拆解到模块、接口、数据流、边界条件层面，AI 才能稳定产出。需求表达本身是一项工程能力。
- **AI 长会话上下文漂移**：跨模块改动时，AI 容易"顺手优化"无关代码或丢失早期约束，需要主动控制任务边界和上下文。
- **跨前后端一致性把控**：AI 容易只改前端展示而漏掉后端权限校验，导致越权问题，必须建立"先画数据流、再写代码"的强制流程。
- **AI 输出的边界条件覆盖**：未登录、未绑定、解绑中、并发、空数据等场景常被 AI 忽略，需要由人主动追问并在 Review 中补齐。

### 项目亮点

- **完整跑通 Vibe Coding 全栈闭环**：作为 5 年前端工程师，主导一个包含小程序前端、NestJS 后端、Prisma 数据模型、Redis 缓存的真实业务系统从 0 到 1 落地，代码全部通过 AI 协同生成。
- **把 AI 从"写代码工具"升级为"工程协作者"**：AI 参与了需求拆解、方案对比、影响面分析、初稿生成、Bug 定位、文档撰写，我承担需求表达、架构决策、Review、验收。
- **共享厨房模型的全栈权限设计**：通过引导 AI 在后端统一收口"厨房用户范围"（`resolveKitchenUserIds`），把菜谱可见性、购物车、订单创建的校验复用同一作用域，避免"前端隐藏但接口可绕过"的越权问题。
- **沉淀可复用的 Vibe Coding 方法论**：六阶段工作流 + Trellis 任务上下文管理 + AI 输出 Review 清单，可直接迁移到团队协作场景。

在这个项目中：

- **AI 做了什么**：根据我的设计和需求描述，生成前端页面、组件、状态管理、HTTP 封装；生成 NestJS 模块、Controller、Service、DTO；生成 Prisma schema、迁移脚本、数据查询；生成 README 和 PRD 初稿。

### 3. 关键业务模型（人工设计 → AI 实现）

#### 双角色模型

- `userType = 0`：饲养员
- `userType = 1`：吃货

不同角色在同一模块下有不同操作：

- 饲养员：创建菜谱、编辑菜谱、处理厨房订单、接单、开始烹饪、完成订单。
- 吃货：浏览菜单、加入购物车、提交订单、查看我的订单、取消订单。

#### 共享厨房模型

共享厨房的核心是"当前用户可见的厨房范围"，由后端统一定义：

- 未绑定：菜谱查询范围为当前用户。
- 已绑定：菜谱查询范围为当前用户 + 绑定对象。
- 创建菜谱：仍归属当前用户。
- 编辑/删除菜谱：只允许创建者操作，由后端返回 `canManage` 标记驱动前端入口。
- 加购物车 / 下单：后端校验菜谱是否属于当前共享厨房。

这套模型的关键决策由我做出：**不引入新数据表，而是在后端运行时动态计算厨房成员**。这样既避免数据迁移成本，又让解绑后的数据隔离更自然。AI 负责按这个决策落地代码（`resolveKitchenUserIds` 等）。

#### 订单状态模型

订单状态由后端统一维护：待接单 → 已接单 → 烹饪中 → 已完成 / 已取消。

前端根据角色和状态展示按钮，后端做最终状态流转校验，避免前端绕过。

## 五、Vibe Coding 工作流（核心）

> 这是简历和面试中最适合突出 AI 经验的部分。
>
> 核心理念：**AI 是协作者而不是替代者；工程师对最终代码、业务正确性和跨层一致性负全责。**

### 1. 协作模型：我说想法 → AI 实现 → 我把控

```text
┌─────────────────────┐         ┌─────────────────────┐
│   我（工程师）       │         │      AI             │
├─────────────────────┤         ├─────────────────────┤
│ • 产品设计          │ ──需求──▶│ • 需求拆解          │
│ • 业务建模          │ ──约束──▶│ • 代码阅读          │
│ • 技术决策          │ ──取舍──▶│ • 方案候选          │
│ • Review 与验收     │◀──产出── │ • 代码生成          │
│ • 联调验证          │ ──问题──▶│ • Bug 定位          │
│ • 知识沉淀          │◀──初稿── │ • 文档撰写          │
└─────────────────────┘         └─────────────────────┘
```

**关键约定**：

- 所有需求先由我拆解到模块、数据流、边界条件层面，再交给 AI。
- 所有跨层改动，AI 必须先输出影响面清单，由我确认后才能写代码。
- 所有 AI 生成代码必须经过我的 Review，至少跑一项验证（typecheck / build / 真机 / 接口）。
- 所有架构决策（加表 vs 不加表、前端校验 vs 后端校验）由我做，AI 不能擅自升级架构。

### 2. 总览：六阶段工作流

```text
需求描述
    ↓
[1] 需求澄清（AI 拆解 + 人工取舍）
    ↓
[2] 代码阅读（AI 汇总调用链 + 人工补盲）
    ↓
[3] 方案设计（AI 给选项 + 人工决策）
    ↓
[4] 代码生成（AI 写初稿 + 人工 Review）
    ↓
[5] 联调验证（typecheck / build / 真机 / 数据校验）
    ↓
[6] 文档沉淀（README / PRD / 任务记录 / 项目规范）
```

### 3. 阶段一：需求澄清——把一句话拆成可执行任务

项目中很多需求最初都是自然语言，例如：

> 两个用户绑定后，他们分别创建的菜谱应该组合成一个厨房，双方都能看到和下单。

**AI 做什么**：

- 总结需求目标、用户角色、业务边界和可能影响的模块。
- 区分"可从代码推导的信息"和"需要产品决策的问题"。
- 把需求拆成前端展示、后端查询、权限校验、数据恢复、验收标准等子任务。

**我做什么**：

- 判断哪些是 MVP 必须做，哪些是后续扩展。
- 决定是否引入新数据表，还是复用现有情侣关系动态计算厨房范围。
- 明确编辑/删除权限是否开放给对方，避免 AI 过度扩展。

### 4. 阶段二：代码阅读——先理解再动手

跨模块改动前，先让 AI 阅读相关文件，而不是直接生成实现：

- 前端：`pages/cart/component/CateTab.vue`、`api/recipes.js`、`stores/recipe.js`
- 后端：`recipes.service.ts`、`cart.service.ts`、`orders.service.ts`、`schema.prisma`

**AI 做什么**：汇总当前数据流、标记按单用户查询的接口、找出可能绕过前端的后端入口。

**我做什么**：判断 AI 给出的影响面是否完整，主动补充遗漏路径（例如"订单创建时传 cartList 也要校验菜谱归属"），避免只改 UI 不改后端。

### 5. 阶段三：方案设计——AI 给选项，我做决策

共享厨房功能曾有多种候选方案：

- 方案 A：新增 `Kitchen` 表，绑定时迁移菜谱归属。
- 方案 B：不改表结构，按情侣关系动态计算厨房成员（运行时作用域）。
- 方案 C：菜谱仍归属个人，但接口层根据情侣关系合并查询范围。

**AI 做什么**：对比实现成本、迁移成本、解绑语义、扩展空间，给出推荐和风险。

**我做什么**：选择 MVP 阶段更稳妥的方案 B（动态作用域），不因为 AI 倾向"架构化"就过早加表。这种取舍能力是工程师的核心价值。

### 6. 阶段四：代码生成——AI 写初稿，我负责边界和风格

AI 在我引导下生成的典型代码：

- 后端 `resolveKitchenUserIds` 作用域查询。
- 前端 `canManage` 控制编辑/删除入口的展示。
- Prisma schema 的字段补充和迁移。
- 订单状态流转的 Service 实现。
- 前端 Pinia store 的状态合并逻辑。
- README、PRD、简历项目文档的初稿。

**AI 初稿绝不直接合并。** 我的 Review 重点：

| Review 维度 | 检查内容 |
| --- | --- |
| 风格一致 | 命名、目录、引入路径、错误处理是否贴合现有代码 |
| 范围控制 | 是否误改无关代码、是否"顺手优化"未在需求中的内容 |
| 边界条件 | 未登录、未绑定、解绑中、并发、空数据是否覆盖 |
| 跨层一致 | 前端展示、后端查询、权限校验、数据库约束是否对齐 |
| 类型安全 | TypeScript 类型、Prisma 类型、DTO 校验是否完整 |
| 可验证性 | 是否能跑过 typecheck / build / 真机 / 接口测试 |

### 7. 阶段五：联调验证——不把"AI 说可以"当结果

每次实现后按功能类型验证：

- 前端文档：`git diff --check`
- 后端服务：`pnpm typecheck`、`pnpm build`
- 数据校验：Docker MySQL / Redis 命令做只读核对
- 小程序体验：微信开发者工具 + 真机预览
- 跨层链路：从页面操作到接口、数据库、状态回写逐层核对

这一步避免 AI 代码"看起来对、但业务链路不闭环"。

### 8. 阶段六：文档沉淀——把协作过程变成资产

项目沉淀了：

- 前端 README、后端 README
- 共享厨房 PRD
- Trellis 任务记录（每个功能/Bug 都有独立任务目录、PRD、验收标准）

## 七、项目难点与解决思路

### 1. 模糊需求 → AI 可执行任务的转化

AI 对模糊需求的处理质量很差。一句"双方菜谱共享"可以有十种实现方式，AI 可能会随机挑一种。

**解决方式**：在交给 AI 之前，先由我把需求拆解到"模块 + 数据流 + 边界条件 + 验收标准"四要素，AI 才能稳定产出。

### 2. AI 长会话上下文漂移

跨模块改动时，AI 容易"顺手优化"无关代码，或者忘记早期约束。

**解决方式**：

- 使用 Trellis 任务目录隔离每个功能的会话上下文。
- 每次任务前明确边界："只改 X 文件、不动 Y 模块"。
- 关键约束在每轮对话中重申，避免 AI 漂移。

### 3. 跨前后端一致性把控

AI 容易只改前端展示而漏掉后端权限校验，造成"前端隐藏但接口可绕过"的越权问题。

**解决方式**：

- 跨层改动前，强制 AI 先输出调用链清单。
- 后端统一收口权限作用域（`resolveKitchenUserIds`），所有相关接口复用。
- 前端只做展示控制，权限以后端为准。

### 4. AI 输出的边界条件覆盖

未登录、未绑定、解绑中、并发、空数据等场景常被 AI 忽略。

**解决方式**：

- Review 时按"边界条件清单"逐项追问。
- 把高频遗漏点写进项目规范，AI 下次会主动覆盖。

### 5. 真机调试细节

接口地址、安全区、自定义导航、弹窗遮罩、图片上传等问题在真机和模拟器之间表现不一致。

**解决方式**：`BASE_URL` 支持局域网/内网穿透切换；封装安全区和导航高度工具；所有体验问题在真机复测。

## 九、面试问答建议

### Q1：这个项目是你自己写的代码吗？

> 诚实地说，**项目代码主体是我通过 Vibe Coding 让 AI 生成的**。我作为 5 年前端工程师，承担的是产品设计、业务建模、需求表达、技术决策、AI 输出 Review、跨前后端联调和质量验收的全流程主导工作。我把这种模式叫做"Vibe Coding 全栈实践"——它不是"AI 帮我写一段代码"，而是"我用工程化方法主导 AI 完成一个真实的全栈业务系统"。这种模式对我的能力要求不是变低，而是从"亲手写代码"换成了"指挥 AI 写出能用、能跑、可维护的代码，并对结果负全责"。

### Q2：那你在这个项目里的真实价值是什么？

> 我的价值在四个层面：第一，**产品和业务设计**——把"情侣厨房互动"这种模糊概念拆成可落地的角色、绑定、共享厨房、订单状态机；第二，**技术决策**——比如共享厨房选择动态作用域而不是加表，是我做的取舍；第三，**质量把控**——AI 生成的每一段代码都要经过我对风格、边界、跨层一致性的 Review，至少跑一项验证；第四，**工作流设计**——我设计了六阶段流程和任务上下文管理机制，让 AI 协作可重复、可验证。AI 把我的"思考成本"转成了"判断成本"，而判断能力来自多年工程经验。

### Q3：项目里最有技术含量的点是什么？

> 是共享厨房和订单链路。它涉及情侣绑定关系、菜谱可见范围、创建者权限、购物车添加、订单提交多个层级。我设计的方案是在后端抽象厨房用户集合（`resolveKitchenUserIds`），所有菜谱可见性和下单校验复用同一作用域，前端只根据 `canManage` 控制操作入口。AI 负责把这个设计落地到前后端代码。这个方案的核心是"权限边界统一在后端、前端只做展示"——这是我做的架构决策，避免了 AI 倾向于在前端做权限校验的常见陷阱。

### Q4：你怎么保证 AI 生成代码的质量？

> 我有一套六阶段工作流和 Review 清单。具体抓手包括：**范围控制**——每次只让 AI 改一个功能点，禁止顺手优化；**风格一致**——让 AI 先读同类文件再生成；**影响面**——跨模块改动必须先输出调用链清单，我确认后才写代码；**边界条件**——主动追问未登录、未绑定、解绑、并发、空数据等场景；**验证手段**——typecheck、build、真机、接口抓包至少跑一项；**上下文管理**——用 Trellis 任务目录隔离不同功能的会话，避免 AI 长会话漂移。原则是：AI 写的每一行都要能在一个明确的验证手段下被复核。

### Q5：作为前端，你怎么把控 AI 写的后端代码？

> 关键不在于"我能不能亲手写后端"，而在于"我能不能判断后端代码是否合理"。这个项目里我能做到：第一，**看懂 NestJS 的 Controller / Service 分层和 Prisma schema**，能识别接口职责是否单一、字段建模是否合理；第二，**把控权限边界**——什么校验必须放后端、什么数据流不能让前端绕过；第三，**通过接口抓包和数据库读校验来验证 AI 输出**，而不是只看代码"看起来对不对"。坦白说，我亲手写后端会比 AI 慢得多，但我能保证 AI 写的后端在业务上是对的、在权限上是闭合的。

### Q6：在团队里怎么落地这套 Vibe Coding 工作流？

> 三个层面：第一，**建立任务上下文规范**，每个任务有独立 PRD 和验收标准，让 AI 协作有清晰边界；第二，**建立 Review 清单**，把"风格一致、影响面、边界条件、验证手段"作为 AI 代码进入主干的硬性门槛；第三，**沉淀团队级提示词和模式库**，把高频踩坑点写成 AI 可读的项目规范，让团队所有人复用同一份上下文。这套方法不是要替代 Code Review，而是把 Review 的重点从"语法和风格"转移到"业务正确性和跨层一致性"。

### Q7：你觉得 Vibe Coding 会让前端工程师被替代吗？

> 不会替代，但会重新分配价值。AI 替代的是"把已知方案敲成代码"的部分，但产品理解、业务建模、技术取舍、质量判断、跨团队协作仍然需要工程师。Vibe Coding 实际上对工程师的要求更高——你需要既懂产品又懂技术，既能拆解需求又能 Review 代码，既能做架构决策又能端到端联调。这个项目就是我在验证：5 年前端经验在 Vibe Coding 模式下能交付什么样的产品。
